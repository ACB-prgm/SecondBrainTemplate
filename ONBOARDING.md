# Onboarding

Use this when the user explicitly asks to be onboarded, says they are new, asks what SecondBrain can do, or when no private `SecondBrain` repository exists.

## Goal

Onboarding should get the user to perceived value quickly. Do not optimize for the fewest steps; optimize for the best friction-to-value ratio.

## Principles

- Sell outcomes, not features.
- Show one useful result as soon as possible.
- Personalize from the user's stated goal or provided context.
- Use progressive disclosure instead of a long upfront tutorial.
- Keep prompts short and action-oriented.
- Stop onboarding once the user is already getting value.

## Flow

1. Briefly explain that SecondBrain stores structured personal knowledge in a private GitHub repo.
2. Ask what the user wants help with first using a short menu.
3. Recommend one first action based on their answer.
4. Use the normal action approval flow for any external write.
5. Summarize what was created, stored, linked, or configured.

Suggested menu:

1. Store and retrieve personal records.
2. Track follow-ups and unresolved questions.
3. Add tasks, reminders, or calendar events.
4. Store growing tabular data like expenses or inventory.
5. Learn the repo structure and maintenance workflow.

## First Useful Result

Try to create an immediate useful result, such as:

- saving a small record and returning its retrieval handle,
- identifying follow-ups from pasted information,
- creating a task or calendar event from concrete user-provided details,
- saving a durable user preference after explicit approval,
- showing how a stored item can be found later.

Do not ask the user to design the whole system before they get value.
