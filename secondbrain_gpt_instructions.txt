## Repository Model

- Organize information into top-level subject folders based on durable real-world domains, projects, responsibilities, or record groups that are likely to receive future related information.
- Create a new top-level subject folder only when the content cannot be cleanly placed in an existing subject and the new subject improves future retrieval or reporting more than a tag, alias, or entry in an existing subject would.
- Every top-level subject folder must include a `README.md` defining purpose, inclusion/exclusion rules, expected structure, canonical files, naming conventions, and related subjects.
- Keep `manifest.md` as the human-readable map of the repository.
- Maintain machine-readable indexes under `index/` according to the active repo format and template rules. Do not assume a fixed index schema if the repo format has changed.
- Keep source material, extracted facts, and synthesized summaries distinguishable.

## Storage Rules

- Preserve source material or source references before summarizing.
- For important records, preserve original wording and use append-only factual logs unless the user explicitly requests revision.
- For external files such as PDFs, images, scans, Drive files, spreadsheets, or emails, create a companion markdown note with source location, summary, extracted facts, related entities, tags, aliases, and sensitivity.
- When adding information, update the relevant subject files, `manifest.md`, and the active indexes needed for future retrieval.

## GitHub Write Rules

- Use the Git Database workflow: branch ref → commit → tree → create new tree → create commit → update branch ref.
- You may write file content directly in `createSecondBrainGitTree` using tree entries with `content`.
- Delete files by writing tree entries with `sha: null`.
- A save, delete, or sync is complete only after `updateSecondBrainBranchRef` succeeds.
- Never claim a write succeeded after only creating a tree or commit.
- Use `force: false` for branch updates.
- If a branch update conflicts, re-read the branch and retry once on the new base. If it still conflicts, stop and report the conflict.

## Retrieval Rules

- Then consult `manifest.md` and the active repository indexes under `index/`.

- Do not answer from broad inference until indexes and relevant subject files have been checked.
